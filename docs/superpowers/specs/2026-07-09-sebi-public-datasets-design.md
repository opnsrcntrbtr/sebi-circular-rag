## Goal

Extract the public-domain SEBI regulatory data held in this repository into structured, documented datasets for AI/ML research, packaged for publication on Hugging Face, Kaggle, Zenodo, and IndiaAI/AIKosh.

## Dataset Design — repo `sebi-circulars` (multi-config)

### Config 1: `corpus` (flagship)
One row per circular, direct transform of `circulars.jsonl`.
- **Schema:** `circular_number (str)`, `issue_date (date)`, `effective_date (date, nullable)`, `subject (str)`, `issuing_department (str)`, `supersession_status (str: in_force|superseded|amended)`, `version_lineage (list[str])`, `source_url (str)`, `text (str)`, `excerpt (bool)`.
- **Transforms:** drop `provenance` (local filenames, no research value) or rewrite to `extraction_date`; empty strings → null; validate with existing `scripts/validate_corpus.py` invariants before export.
- **Formats:** Parquet (HF viewer-friendly) + JSONL (Kaggle/AIKosh).

### Config 2: `chunks`
One row per retrieval chunk.
- **Schema:** `chunk_id`, `doc_id`, `section`, `text`, plus flattened metadata columns (dedupe the nested `meta` object into top-level columns).
- **Transform:** strip the repeated contextual header from `text` into its own `context_header` column so researchers can choose either representation.

### Config 3: `lineage`
Edge list derived from forward maps only (`superseded_by`/`amended_by` are inverses — regenerate, don't duplicate).
- **Schema:** `source_circular`, `relation (supersedes|amends)`, `target_circular`, `source_issue_date (nullable)`, `target_in_corpus (bool)`.
- ~1,434 edges over 1,038+ node identifiers. Positioning: temporal regulatory citation graph — supports "is this rule in force?" reasoning tasks, link prediction, and graph-RAG research.

### Config 4: `eval`
Retrieval + abstention benchmark from `golden_v5.jsonl` (56 queries), with a `version` column if earlier versions add non-duplicate queries.
- **Schema:** `id`, `query`, `relevant_circulars (list[str])`, `answer_contains (str)`, `abstain (bool)`, `version`.
- Card must state clearly: small benchmark (n=56), suited to sanity-check evaluation, not leaderboard-grade.

### Config 5: `citation-normalization` (transformed task)
Seq2seq/NER-style pairs: raw in-text reference string → normalized circular number. Generated by re-running the hardened extraction strategies (`src/sebi_rag/ingest_pdf.py`, regression-tested by `tests/test_ingest_refs.py`) over corpus text and emitting matched span + normalized form.
- **Schema:** `raw_reference`, `normalized_circular_number`, `context_window (str)`, `source_doc_id`, `format_family (str)` (the 7 validated format families).
- Value: SEBI reference formats are messy (space-split, dept-only, HO/(N)YEAR-DEPT, 2026 departmental orders) — a real-world string-normalization task the repo's parser already solves, so labels are high quality.

### Config 6: `supersession-pairs` (transformed task)
Pair-classification set: `(circular_a_number, circular_a_subject, circular_b_number, circular_b_subject, label ∈ {supersedes, amends, unrelated})`. Positives from lineage edges (~1,434); negatives sampled from same-department non-linked pairs at ~2:1. ~4–5k rows.

### Deferred (documented, not built): `embeddings`
Precomputed BGE-M3 vectors keyed by `chunk_id` (~150 MB Parquet). Reproducible from `chunks`; include only if there's demand — keeps the repo lean.

## Licensing & Compliance

- **Underlying text:** SEBI circulars are Indian government works (Copyright Act 1957 §52(1)(q) permits reproduction of government orders/notifications). Attribute SEBI, retain `source_url` per record.
- **Compilation/annotations** (metadata extraction, lineage graph, chunking, task labels): **CC-BY-4.0**.
- **HF license field:** `cc-by-4.0` with a "Source Data" card section explaining the government-work status of the underlying text.
- **Disclaimers (all cards):** not legal advice; not affiliated with or endorsed by SEBI; verify against sebi.gov.in before regulatory reliance; corpus covers 2021–2026 and is not exhaustive of all SEBI circulars.
- **AIKosh alignment:** data is non-personal public regulatory text — meets their non-personal/anonymized requirement; complete metadata + licensing doc satisfies their submission review.

## Extraction Architecture (for the implementation phase)

Approaches considered:
- **A (chosen): single export script** — `scripts/export_datasets.py` reading `data/corpus`, `data/index`, `eval/golden` and writing `dist/datasets/<config>/` (Parquet + JSONL) plus generated dataset cards and SHA-256 manifest. Add `make export-datasets`. Simple, reproducible, testable with the existing offline test suite pattern.
- **B: HF `DatasetBuilder` loading script** — richer typing but HF has deprecated script-based datasets in the viewer; Parquet-native is the current best practice. Rejected.
- **C: DVC/versioned data pipeline** — overkill for a 14 MB corpus with an existing reindex pipeline. Rejected.

Pipeline stages: **validate** (reuse corpus validation invariants) → **transform** (per-config builders, pure functions, unit-testable) → **emit** (Parquet/JSONL/CSV + cards + checksums) → **package** (platform-specific bundles: HF folder layout, Kaggle zip, Zenodo tarball, AIKosh pack). Uploads are separate, explicitly user-triggered steps.

Versioning: dataset version tied to corpus snapshot date (e.g. `v2026.07`); manifest records source-file checksums so future corpus growth (603 → more) produces clean incremental releases.

## Testing

- Unit tests per config builder: schema conformance, row counts vs source, no empty `text`, lineage edge count parity, citation-pair labels round-trip through `normalize_circular_number`.
- Golden validation: exported `corpus` config re-passes `validate_corpus` invariants.
- Card lint: YAML config block parses; license/disclaimer sections present.
